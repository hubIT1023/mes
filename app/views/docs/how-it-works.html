How the Maintenance System Works

Welcome to the HubIT Maintenance Management System technical guidelines. This reference manual clarifies the architectural flows, background automation schedulers, and secure SSL processes running under the hood.

System Architecture Operation Workflows Download PDF Manual

1. System Architecture Layout

Structured using clean, highly readable PHP MVC blueprints.

MVC Architecture

Separates visual displays (Views), business rules (Models), and processing logic (Controllers) to guarantee maintainability:

  • Controllers: Bind requests, validate parameters, and render views.
  • Models: Safely interface database connections using parameterized statements.
  • Views: Pure PHP layouts injected with dynamically escaped data keys.

Dynamic Routing Engine

The front controller (index.php) handles incoming URLs dynamically:

  • Base Path Detection: Auto-detects whether the URL is served from root (/) or sub-folder (/mes/).
  • Safe Redirects: Ensures references do not break when porting between local environments and cloud servers.

Multi-Tenant Isolation

Keeps client organization records strictly confidential:

  • Session Scoping: Reads tenant_id straight from validated server sessions.
  • Where Scopes: Parameterizes every database SELECT/UPDATE statement with the current organization UUID context.

2. Core Operational Workflows

Step-by-step processes managing daily asset tasks.

Flow A: Time-Based Preventive Maintenance (TBPM) Schedulers

01

Cron Trigger Execution

The system's hosting environment schedules a background cron job triggering the CLI script app/cron/tbpm_scheduler.php.

02

Due Maintenance Selection

The scheduler loops through all registered tenant orgs, mapping configs, and querying the database to find assets whose next_maintenance_date matches today's date.

03

Work Order Generation

Creates an active entry in the routine_Work_Orders table matching the predefined checklist template, setting the initial status to "Pending".

04

Interval Date Reprojection

Calculates the next maintenance date based on the config interval (e.g. adding 30 or 90 days), updating the record context automatically.

Flow B: Technician Checklist Execution & Archival

01

Begin Maintenance

Technicians log into their dashboard portal, see active pending schedules, and click "Associate Checklist". This updates the work order status to "On-Going".

02

Real-Time Data Capture

The technician logs findings, notes, checkmarks, and custom measurement metrics directly on the checklist items.

03

Atomic Database Save

When clicking 'Save Progress', the controller fires updateChecklistWithTasks(). This updates both the header and child task tables atomically inside a secure SQL transaction.

04

Archival & Cleanup

Once finalized, the system creates a copy in the completed history records (retaining logs forever for auditing) and safely deletes the checklist instance from active tables to keep operations clean.

3. Automated Let's Encrypt Updates

Resolving SSL boot issues with zero manual configuration.

Solving the Chicken-and-Egg SSL Dilemma

Apache servers crash on boot if their virtual host SSL certificates are missing. But to request certificates from Let's Encrypt via Certbot, Apache must be active on port 80 to verify domain ownership.

The system's new startup script, docker-entrypoint.sh, solves this loop automatically:

Boot Step 1
Valid Let's Encrypt Verification

The container checks the issuer of local certs. If missing or self-signed, it generates dummy credentials so Apache starts safely.

Boot Step 2
Background Certbot Loop

A background daemon waits for Apache, requests real certificates from Let's Encrypt using webroot, swaps them, and reloads Apache gracefully.

• Daemon Active

12-Hour Renewal Check

A recurring loop inside the container tests certificate expiration. Every 12 hours, it attempts to renew, triggering a reload only if updates succeed.

Trigger Command: certbot renew --webroot -w /var/www/html
